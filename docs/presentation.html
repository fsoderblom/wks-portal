The Problem

Your organisation uses OpenPGP for encrypted email. How do senders find recipients' public keys?

Key servers

  • No ownership validation
  • Poison attacks
  • GDPR problems
  • Unavailable sporadically

Manual process

  • IT ticket per user
  • Error-prone
  • Slow (days lag)
  • No self-service

WKD + WKS

  • Domain-controlled key discovery via HTTPS
  • Email ownership proof
  • HTTPS delivery
  • Self-service portal
The classic key server model has well-known problems. Key poisoning, no identity validation, GDPR headaches. The alternative — manual IT workflows — doesn't scale. WKD and WKS, standardised by the IETF and GnuPG project, solve this elegantly. (~2 min)

WKD and WKS Explained

WKD — Web Key Directory

  • Keys served over HTTPS
  • URL derived from email address
  • Looked up by email client automatically
  • https://openpgpkey.example.com/
    .well-known/openpgpkey/
    example.com/hu/<hash>

WKS — Web Key Service

  • Protocol for key submission
  • Challenge/response via email
  • Proves mailbox ownership
  • Send key → receive challenge
    → reply → key gets published

OpenPGP: RFC 9580 · WKD/WKS: draft-koch-openpgp-webkey-service · GnuPG gpg-wks-server

WKD is the read side — any email client that understands it can look up a key automatically. WKS is the write side — it's how keys get published, using email as a proof of ownership mechanism. GnuPG ships a reference server implementation. We built the enterprise wrapper around it. (~1.5 min)

Identity is asserted via X-User from a trusted reverse proxy — Flask never handles credentials

nginx sets: auth_request_set $email $upstream_http_x_auth_request_email;
proxy_set_header X-User $email;
Authentication is handled entirely at the proxy layer. oauth2-proxy validates the OIDC token, nginx extracts the verified email claim and passes it as X-User. Flask trusts this header completely — it never sees passwords or tokens. This is a well-established pattern for internal enterprise apps. (~1.5 min)

Email Backend — Postfix Transport

transport table

key-submission@domain.cc  wks:

master.cf addition

wks unix - n n - - pipe
  flags=Rq user=webkey
  argv=/usr/local/sbin/wks-receive

wks-receive (11 lines)

#!/bin/sh
exec /usr/bin/gpg-wks-server -v \
  --receive \
  --send \
  --from key-submission@domain.cc \
  --header X-WKS-Loop=wks.domain.cc

Delegates all WKS protocol logic to GnuPG

The email path is beautifully minimal. Postfix routes key-submission@ to a pipe transport. The pipe is literally 11 lines of shell — it just calls gpg-wks-server. All protocol complexity is in GnuPG. This is the "delegation over reimplementation" principle. (~1.5 min)

→

↓

→
Confirmation link → /confirm?token=<uuid>
↓
5. User clicks
→

The confirmation flow is the core of the system. It's a simple state machine — upload, stage, prove ownership via email click, publish. The key insight is that we never trust user input past the initial validation: keys are staged in a restricted directory and only published after the email confirmation proves mailbox ownership. (~2 min)

def parse_key(asc_path):

            [GPG, '--homedir', tmpdir,
             '--import-options', 'show-only',
             '--import', asc_path],
            capture_output=True, timeout=10
        )
        # extract fingerprint and email UIDs from stdout
        ...
    finally:
        shutil.rmtree(tmpdir, ignore_errors=True)  # always clean up

Isolation

  • Fresh homedir per request
  • Key never imported globally
  • Temp dir always removed

Validation

  • Max 512 KB upload
  • Domain whitelist check
  • Dual regex UID extraction

Publication

  • Only via --install-key
  • After email confirmation
  • Single-use token
This is the part I'm most proud of. When a user uploads a key, we never import it into any persistent keyring. We spin up a throwaway GPG homedir, parse the key there, and immediately delete the directory — even if an exception occurs. This prevents key contamination and limits blast radius from malicious inputs. (~2 min)

Security Architecture: Defence in Depth

LayerControlMitigates
Network allow 10.0.0.0/8; deny all; Unauthenticated internet access
Identity OIDC via oauth2-proxy Anonymous submissions
Email proof Single-use UUID token, 60 min TTL Publishing keys for addresses you don't own
Key parsing Isolated temp GPG homedir Malicious key / keyring poisoning
SQL Parameterised queries only SQL injection
Process Unprivileged webkey + systemd NoNewPrivileges, PrivateTmp, ProtectSystem=strict Privilege escalation, filesystem exposure
TLS Mozilla Intermediate cipher suite, TLS 1.2+1.3 only Weak cipher negotiation
/admin/ nginx IP allowlist + WKS_PORTAL_ADMIN_USERS identity check Unauthorised key deletion
HTTP X-Content-Type-Options, Referrer-Policy MIME sniffing, referrer leakage
The security model is defence in depth — each layer catches a different class of problem. No single control failure compromises the system. I'll highlight the most interesting ones. (~2 min)

Token Design

Database schema (relevant columns)

CREATE TABLE pending (
  id         TEXT PRIMARY KEY, -- UUID
  created_at INTEGER NOT NULL,
  expires_at INTEGER NOT NULL,
  requested_by TEXT,           -- SSO identity
  selected_email TEXT,
  fpr        TEXT NOT NULL,    -- fingerprint
  asc_path   TEXT NOT NULL,
  used       INTEGER NOT NULL DEFAULT 0
);

Confirmation guard

row = db.execute(
  "SELECT * FROM pending
   WHERE id=? AND used=0
   AND expires_at > ?",
  (token, int(time.time()))
).fetchone()
if not row:
    return error("Invalid or expired token")
# mark used immediately
db.execute(
  "UPDATE pending SET used=1 WHERE id=?",
  (token,))

Prevents replay attacks

The token is a UUID — 128 bits of entropy, impossible to guess. It's marked used before the key is published. The SQL check atomically validates expiry, use status, and token authenticity in a single query. (~1.5 min)

Cleanup Strategy

Request-time cleanup

# At the start of every request:
db.execute("""
  DELETE FROM pending
  WHERE expires_at < ?
    OR used = 1
""", (int(time.time()),))

No background worker required

Cron cleanup

# Remove expired WKS requests
25 0 * * * gpg-wks-server --cron

# Remove orphaned .asc files
17 3 * * * find .../pending \
  -name '*.asc' \
  -mtime +2 \
  -size -1M \
  -delete
Cleanup is two-tiered. At the application level, every request triggers a DELETE on expired or used rows — this keeps the database lean without any background job. The cron handles the filesystem side: orphaned .asc files and the WKS server's own state. (~1 min)

Pragmatic Engineering Choices

SQLite over PostgreSQL

  • State is ephemeral (TTL 60 min)
  • Low write volume
  • Zero ops overhead
  • Trade-off: single machine

Delegate to GnuPG

  • gpg-wks-server --install-key
  • gpg-wks-server --receive
  • Protocol correctness guaranteed
  • We build the UX, not the crypto

Python 3.9+ / Flask 2.3+

  • Requires Python 3.9+ and Flask 2.3+
  • Aimed for RHEL 8/9 and Ubuntu 24.04 LTS
  • venv-based install (requirements.txt)
  • RHEL 7 / Python 3.6 preserved on git tag v1-rhel7-compat

Inline HTML templates

  • Single deployable file
  • No template path issues
  • Fully auditable
  • Trade-off: harder to style
These choices reflect a philosophy: build the simplest thing that works in production. We deliberately chose SQLite because the data is transient. We modernised to Python 3.9+ and Flask 2.3+ — the legacy RHEL 7 / Python 3.6 baseline is preserved on a git tag for anyone who still needs it. And we chose to delegate crypto to GnuPG — we're building a UX, not reimplementing OpenPGP. (~2 min)

One nice property of the system is that there are two independent paths to the same published key. Power users and automated tooling can use the direct WKS email protocol. Everyone else gets a nice web UI. The WKD tree is the single source of truth either way. (~1.5 min)

Lessons Learned

Don't reimplement what GnuPG does for you

The WKS protocol has edge cases. Let gpg-wks-server own them. Build the UX wrapper.

Temporary directories are a superpower

Isolated GPG homedirs make untrusted key parsing safe. Always clean up in finally.

Email is your identity proof primitive

The confirmation email isn't just UX friction — it's the ownership proof. Don't skip it.

Match complexity to the problem

SQLite + cron instead of Postgres + Celery. Inline HTML instead of React. Right tool, right scope.

These are the things I'd tell someone starting a similar project. The biggest win was treating GnuPG as a black box for the protocol and building only the enterprise wrapper. The biggest risk we avoided was overbuilding — this system has been running in production with minimal ops burden. (~2 min)
